# 数据库设计说明（V1）

## 一、项目说明

本系统为简化版开票管理系统，用于实现基础的开票业务流程，包括：

- 用户管理
- 客户管理
- 商品管理
- 销售单创建
- 销售单明细记录
- 历史销售单查询
- 打印销售单

系统采用前后端分离架构设计，数据库用于支撑核心业务数据存储。

---

## 二、设计目标

本数据库设计遵循以下原则：

1. 结构清晰，支持快速开发
2. 满足当前简化业务需求（MVP）
3. 保留必要扩展能力
4. 避免过度设计
5. 支持后期功能扩展（如商品分类、用户权限等）
## 二、数据表清单

本系统包含以下核心数据表：

### 1. 用户表（user）
用于系统登录与基础用户管理。

---

### 2. 客户表（customer）
用于记录开票对象信息，用于销售单归属。

---

### 3. 商品表（product）
用于维护可开票商品基础信息。

---

### 4. 销售单主表（invoice）
用于记录一次完整的开票行为。

---

### 5. 销售单明细表（invoice_item）
---

## 四、用户表（user）

### 4.1 表作用

用户表用于系统登录与身份识别，是整个系统的基础表。

---

### 4.2 字段设计

| 字段名 | 类型 | 是否必须 | 说明 |
|--------|------|----------|------|
| id | INT | 是 | 主键，自增 |
| username | VARCHAR(50) | 是 | 登录用户名，唯一 |
| password | VARCHAR(255) | 是 | 登录密码（加密存储） |
| created_at | DATETIME | 是 | 创建时间 |

---

### 4.3 设计说明

1. 当前系统仅支持单角色用户（管理员）
2. 暂不引入权限系统（避免复杂化）
3. password 必须预留加密存储能力（不可明文）
4. username 需保证唯一性，用于登录识别

---

### 4.4 未来扩展预留说明

虽然当前不实现，但该表设计已预留扩展能力：

- 后续可扩展 role 字段（用户角色）
- 可扩展 status 字段（账号启用/禁用）
- 可扩展 last_login_time（登录记录）


---

## 五、客户表（customer）

### 5.1 表作用

客户表用于记录开票对象信息，用于销售单归属管理。

在开票系统中，一张销售单必须对应一个客户。

---

### 5.2 字段设计

| 字段名 | 类型 | 是否必须 | 说明 |
|--------|------|----------|------|
| id | INT | 是 | 主键，自增 |
| name | VARCHAR(100) | 是 | 客户名称 |
| phone | VARCHAR(20) | 否 | 联系电话 |
| address | VARCHAR(255) | 否 | 地址信息 |
| created_at | DATETIME | 是 | 创建时间 |

---

### 5.3 设计说明

1. 客户是销售单的归属对象，用于区分不同开票记录
2. 当前系统不做客户等级、欠款等复杂业务
3. phone 和 address 为可选字段，避免初期数据强制复杂化

---

### 5.4 未来扩展预留说明

该表预留以下扩展方向：

- 客户等级（普通客户 / VIP客户）
- 客户信用额度
- 客户对账信息
- 客户历史消费统计    


---

## 六、商品表（product）

### 6.1 表作用

商品表用于维护系统中可用于开票的商品基础信息。

商品是销售单的核心组成部分之一。

---

### 6.2 字段设计

| 字段名        | 类型           | 是否必须 | 说明    |
|------------|--------------|----------|-------|
| id         | INT          | 是 | 主键，自增 |
| name       | VARCHAR(100) | 是 | 商品名称  |
| spec       | VARCHAR(100) | 否 | 商品规格  |
| created_at | DATETIME     | 是 | 创建时间  |
| ref_price  |              | 是 | 参考价钱  |
| ref_price  |              | 是 | 单位    |
### 6.3 设计说明

1. 当前仅保留商品基础信息，用于开票选择
2. 不引入库存管理（系统不涉及仓储逻辑）
3. 不引入价格字段，价格在开票时填写（避免价格变动影响历史数据）
4. 商品信息以“静态信息”为主

---

### 6.4 关键设计思想（非常重要）

本系统采用“开票快照原则”：

- 商品信息用于选择
- 实际价格由销售单记录
- 不依赖商品当前状态

---

### 6.5 未来扩展预留说明

该表未来可扩展方向包括：

- 商品分类（category）
- 商品零售价体系
- 商品图片
- 商品上下架状态
- SKU体系（多规格管理）


---

## 七、销售单主表（invoice）

### 7.1 表作用

销售单主表用于记录一次完整的开票行为，是整个开票流程的核心数据载体。

每一张销售单对应一次开票记录。

---

### 7.2 字段设计

| 字段名 | 类型 | 是否必须 | 说明 |
|--------|------|----------|------|
| id | INT | 是 | 主键，自增 |
| customer_id | INT | 是 | 客户ID |
| total_price | DECIMAL(10,2) | 是 | 总金额 |
| created_at | DATETIME | 是 | 创建时间 |

---

### 7.3 设计说明

1. invoice 是开票系统的核心业务表
2. 每张销售单必须绑定一个客户（customer_id）
3. total_price 由系统自动计算（来自明细表汇总）
4. 该表只记录“单据级信息”，不存商品明细

---

### 7.4 数据关系说明

- customer (1) —— (N) invoice
- invoice (1) —— (N) invoice_item

---

### 7.5 关键设计思想

本系统采用“主从结构设计”：

- invoice：记录订单整体信息
- invoice_item：记录具体商品明细

避免将商品信息混入主表，提高结构清晰度

---

### 7.6 未来扩展预留说明

该表未来可扩展：

- 状态字段（已保存 / 已打印 / 作废）
- user_id（操作人）
- invoice_no（业务单号）
- payment_status（支付状态）

---

## 八、销售单明细表（invoice_item）

### 8.1 表作用

销售单明细表用于记录每一张销售单中包含的具体商品信息。

一张销售单可以包含多个商品，因此该表用于实现“一对多结构”。

---

### 8.2 字段设计

| 字段名 | 类型 | 是否必须 | 说明 |
|--------|------|----------|------|
| id | INT | 是 | 主键，自增 |
| invoice_id | INT | 是 | 销售单ID（外键关联 invoice） |
| product_id | INT | 否 | 商品ID（可选，用于扩展查询） |
| product_name | VARCHAR(100) | 是 | 商品名称（快照） |
| spec | VARCHAR(100) | 否 | 商品规格（快照） |
| price | DECIMAL(10,2) | 是 | 单价 |
| quantity | INT | 是 | 数量 |
| subtotal | DECIMAL(10,2) | 是 | 小计金额 |

---

### 8.3 设计说明

1. invoice_item 是 invoice 的从表，用于存储商品明细
2. 使用 invoice_id 作为关联字段，实现一对多关系
3. 商品信息采用“快照存储”原则（product_name / spec）
4. 即使商品后续修改，也不会影响历史订单数据
5. subtotal = price × quantity，由后端计算生成

---

### 8.4 数据关系说明

- invoice (1) —— (N) invoice_item
- product (1) —— (N) invoice_item（可选关联，用于扩展查询）

---

### 8.5 关键设计思想

本表采用“双层策略”：

- 业务关联：通过 product_id（可选）
- 历史稳定：通过 product_name + spec 快照

保证：
- 数据可追溯
- 历史订单不受商品变更影响

---

### 8.6 未来扩展预留说明

该表未来可扩展：

- 折扣字段（discount）
- 税率字段（tax）
- 批次号（batch_no）
- 商品图片快照
## 四、设计原则

1. 所有业务必须可扩展
2. 销售单采用“主表 + 明细表”结构
3. 商品信息允许未来扩展字段（如分类、价格）
4. 不做库存管理
